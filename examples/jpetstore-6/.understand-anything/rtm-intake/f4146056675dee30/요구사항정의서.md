## 2. 개요

- **목적**: 고객 요청 REQ-001("네이버 로그인 기능 추가")을 요구사항 단위로 분해·정의하여 발주-수행 간 범위를 합의한다.
- **범위**: JPetStore 로그인 화면에 네이버 소셜 로그인을 추가하고, 이에 수반되는 OAuth2 연동·계정 연계 데이터·세션 보안을 포함한다. 네이버 외 타 소셜 로그인 확장, 네이버 계정 자체의 회원가입·프로필 관리는 제외한다.
- **용어 정의**: OAuth2 — 네이버가 제공하는 표준 인가 프로토콜. 계정 연계 — 네이버 로그인 사용자와 JPetStore 기존 ACCOUNT 레코드를 매핑하는 것.

| 코드 | 구분 | 설명 |
|---|---|---|

| PER | 성능 요구사항 | 응답시간·처리량 등 |

| QUR | 품질 요구사항 | 신뢰성·사용성·유지보수성 |
| COR | 제약 사항 | 기술·환경적 제약 |

## 4. 요청별 요구사항 정의

### REQ-001 네이버 로그인 추가
- **요청 개요**: 고객이 JPetStore에 네이버 계정으로 로그인할 수 있는 소셜 로그인 기능 추가를 요청함.
- **출처/요청일**: 고객 요청 / 2026-07-19

#### [SFR-010] 네이버 소셜 로그인 — 우선순위: 상 · 상태: 유효
- **정의**: 기존 아이디/비밀번호 로그인과 병행하여, 로그인 폼 내부에서 네이버 계정으로 인증·로그인할 수 있게 하는 기능(사용자 결정: 로그인폼 내부 배치 — Q-2).
- **범위**: 포함 — 로그인 폼 내 네이버 로그인 버튼 노출, 네이버 OAuth 인증 플로우 진입/콜백, 인증 성공 시 기존과 동일한 방식의 세션 구성, 인증 실패 시 기존과 동일한 오류 처리, 이메일 일치 시 기존 계정과 자동 연동(사용자 결정: 기존 계정과 연동 — Q-3). 제외 — 네이버 계정 자체의 회원가입·프로필 관리, 네이버 외 타 소셜 로그인(카카오 등) 확장.
- **출처/관련**: 고객 요청 REQ-001 / 파생 요구사항: SIR-010, DAR-010, SER-010

#### [SIR-010] 네이버 OAuth2 연동 — 우선순위: 상 · 상태: 유효
- **정의**: 네이버 OAuth2 인증서버와 연계하여 인가요청·인가코드-액세스토큰 교환·프로필 조회를 수행하는 인터페이스 요구사항.
- **범위**: 포함 — OAuth2 인가요청/콜백 처리, 인가코드→액세스 토큰 교환, 네이버 프로필 API(닉네임·이메일 등) 조회. 제외 — 네이버 오픈API 쿼터·정책 운영, 다른 소셜 제공자 연동.
- **출처/관련**: 선행 요구사항 SFR-010

#### [DAR-010] 네이버 계정 연계 데이터 — 우선순위: 상 · 상태: 유효
- **정의**: 네이버 로그인 사용자와 기존 로컬 계정(ACCOUNT)을 연계하기 위해 별도 연계 테이블(예: NAVER_ACCOUNT_LINK)을 신설하는 데이터 요구사항(사용자 결정: 별도 연계 테이블 신설 — Q-1).
- **범위**: 포함 — 연계 테이블 신설(ACCOUNT.userid 외래키 + 네이버 고유 식별자), 기존 SIGNON.PASSWORD(NOT NULL) 제약 미변경. 제외 — ACCOUNT/SIGNON 기존 테이블 구조 변경.
- **출처/관련**: 선행 요구사항 SFR-010 / Q-1 답변("별도 연계 테이블 신설") 반영

#### [SER-010] 소셜 로그인 세션 보안 — 우선순위: 중 · 상태: 유효
- **정의**: 네이버 로그인 도입에 따라 기존 인증 상태 판정·세션 관리 원칙을 동일하게 적용하고, 네이버 액세스 토큰을 세션에 원문 노출하지 않도록 보호하는 보안 요구사항.
- **범위**: 포함 — 기존 인증 상태 판정 규칙(인증 표시·계정 정보·아이디 보유) 재사용, 액세스 토큰 세션 원문 미저장 원칙 적용. 제외 — MFA 등 신규 인증 체계 도입.
- **출처/관련**: 선행 요구사항 SFR-010
