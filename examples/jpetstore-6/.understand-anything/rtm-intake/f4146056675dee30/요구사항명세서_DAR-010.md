## 2. 요구사항 상세 명세

| 항목 | 내용 |
|---|---|
| **요구사항ID** | DAR-010 |
| **소속 요청ID** | REQ-001 |
| **요구사항명** | 네이버 계정 연계 데이터 |
| **상태** | 유효 |
| **구분 / 우선순위** | 데이터 / 상(必) |
| **정의** | 네이버 로그인 사용자와 기존 로컬 계정(ACCOUNT)을 연계하기 위해 별도 연계 테이블(NAVER_ACCOUNT_LINK)을 신설하는 데이터 요구사항(사용자 결정: 별도 연계 테이블 신설 — Q-1). |
| **상세 기능** | ① NAVER_ACCOUNT_LINK 테이블을 신설한다(PK: naver_id, FK: userid → ACCOUNT.userid).<br>② 네이버 인증 성공 및 기존 계정 연동 확정 시 매핑 레코드를 저장한다.<br>③ 기존 SIGNON.PASSWORD(NOT NULL) 제약은 변경하지 않는다. |
| **입력 데이터** | 네이버 고유 식별자(naver_id, SIR-010 프로필 조회 결과), 매핑 대상 ACCOUNT.userid |
| **출력 데이터** | NAVER_ACCOUNT_LINK 레코드(naver_id, userid) |
| **처리 흐름** | SIR-010 프로필 조회로 naver_id·이메일 확보 → 이메일로 ACCOUNT 조회해 일치 계정 확인 → NAVER_ACCOUNT_LINK 에 (naver_id, userid) 신규 행 삽입 → 이후 재로그인 시 naver_id 로 연계 조회 |
| **선행 요구사항** | SIR-010 |
| **예외 / 제약** | 동일 naver_id 로 중복 매핑을 시도하는 경우(PK 제약으로 차단) / 매핑된 ACCOUNT.userid 가 삭제될 경우의 FK 무결성 처리 방안 미정([확인필요]) |
| **인수 기준(완료조건)** | ① NAVER_ACCOUNT_LINK 테이블이 정의된 PK/FK 제약대로 생성된다. ② 네이버 최초 로그인 시 매핑 저장 성공률 100%, FK 무결성 위반 0건. |
| **검증 방법** | 검토(스키마 리뷰) + 시험(매핑 저장/조회 시나리오) |
| **출처 / 추적** | SFR-010(네이버 소셜 로그인)에서 파생, Q-1 답변("별도 연계 테이블 신설") 반영 / after-schema.json(NAVER_ACCOUNT_LINK) · jpetstore-hsqldb-data.sql:32,37,38 / 시험케이스 미정([추정]) |
