Code Atlas란?

오픈소스 Understand-Anything(U-A)을 기업 레거시 인수·운영에 맞게 확장한 ktds fork

기반

U-A 코드 이해 엔진

소스 → 지식 그래프(노드·엣지)로 구조화. 기술스택·아키텍처·API·DB를 자동 추출하고 React 대시보드로 시각화.

ktds 확장

결정론 + LLM 2단계

구조(skeleton)는 100% 결정론으로 만들고 LLM은 빈칸(이름·설명·규칙)만 채움. 모든 사실에 파일:라인 인용 의무.

목표

이해 → 변경 → 위임

레거시를 이해하고, 변경 영향을 예측하고, 근거 있는 문서 위에서 AI에게 구현을 위임하는 완결 파이프라인.

U-A가 "구조 그래프"까지라면, ktds는 그 위에 도메인 맵 · 세분화 위키 · 변경 영향도 3개 엔진을 더 쌓았습니다.

ADR-002 · /understand-impact · -review

"이 파일 바꾸면 어디까지?"

map 산출물(edges·ownership·routes) 위에서 역/정 BFS로 영향 범위를 결정론 계산. 재스캔 0회.

예측 — /understand-impact

  • 역 BFS(상류): 영향받는 호출자 → API/배치 진입점
  • 정 BFS(하류): 매퍼·SQL 파일까지 DB 영향 추적
  • 신뢰도 3단계 CONFIRMED / INFERRED / NEEDS_REVIEW
  • 근거 게이트: 구현-임계 사실은 건별 하드차단, 위험은 표면화만

실측 — /understand-review

  • 실제 git diff를 사전 예측과 대조
  • 예측 밖 변경 → 경고, 체크리스트 산출
  • --sr <SR-ID>로 변경 단위 스냅샷 보관
  • 예측 ↔ 실측 2채널 루프로 "계획 vs 현실"

대시보드 오버레이 라벨: 변경예정(시드) · 영향받음 — 노드 배지 + 계층/컨테이너 집계 칩.

ADR-004 · ADR-005 · /understand-docs

거대 단일 파일 → 세분화 위키 메시

5종 근거 문서 + 상태기계

  • 00 기술스택 · 01 아키텍처 · 02 기능명세 · 03 API · 04 DB
  • 신뢰도 4태그 · 파일:라인 근거 · 근거율 투명 보고
  • DRAFT → UNDER_REVIEW → CONFIRMED → APPROVED 승인 흐름
  • 감사 추적 .spec/audit/*.jsonl

세분화 위키(옵시디언 호환)

  • index.md + domain/flow/endpoint/table 노트 + [[위키링크]] 메시
  • 결정론 wiki-graph.json emit (LLM 미경유 → 환각 0)
  • 기본 4계층, --steps로만 step 폭증 구간 opt-in
  • 대시보드 "문서" 뷰 = 리더 레이아웃으로 그대로 소비

ADR-005 AIDD 계약: 참조(R, as-built)와 작업 스토리(W, 변경당)를 하나의 계약으로 통일하되, 개발 방법론(BMad 등)을 교체 가능한 모듈로 분리 — L4 위임구현의 토대.

ADR-003

대시보드는 ktds 소유, 파이프라인은 추종

UI는 자유 개조(영향도·도메인 뷰), 분석 파이프라인은 upstream 개선을 계속 흡수. 충돌점은 workspace·marketplace 2곳뿐.

결정론

동일 입력 → 동일 byte

타임스탬프·랜덤·순서 비결정 금지, 정렬 강제. 재실행 diff=0이 회귀 신호.

근거 우선

인용 없으면 강등

모든 LLM 사실에 파일:라인. 자동 검증 실패 → [확인 필요]. 지어낸 인용 100% 감지.

신호 우선순위

간선 신호 > 파일명 토큰

호출/매퍼 간선이 진실. 파일명 휴리스틱은 폴백일 뿐 — 다음 장의 실데이터 사례가 증명.

흐름 정확도를 끌어올린 4가지

#1 · 엔진 Layer Emit

step이 직접 레이어를 안다

예전엔 대시보드가 파일명으로 레인을 추측 → jpetstore 패키지명 org.mybatis의 "mybatis" 토큰 때문에 전건 DB 오분류. 이제 엔진이 route/edge 근거로 직접 부착.

A · 이벤트별 흐름 분리

ActionBean 1개 → 핸들러별 흐름

Stripes만 클래스당 1흐름으로 뭉쳐 있던 비일관 해소. 이벤트 핸들러마다 라우트 → Account 1 → 7 흐름.

B · 실제 calls 엣지

가짜 합성 순서가 아닌 진짜 호출

step→step을 실제 호출 토폴로지로 emit + 핸들러 bodyText에서 주입 필드 추출해 핸들러별 다른 체인.

C · 곁가지 접기 + 안정화

노이즈 제거 · stale-dist 픽스

추상 베이스 클래스를 step에서 제외. ensure-built가 구 dist를 조용히 실행하던 회귀를 버전 스탬프로 차단.
