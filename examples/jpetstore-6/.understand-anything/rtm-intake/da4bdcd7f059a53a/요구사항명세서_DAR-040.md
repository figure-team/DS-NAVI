## 2. 요구사항 상세 명세

| 항목 | 내용 |
|---|---|
| **요구사항ID** | DAR-040 |
| **소속 요청ID** | REQ-003 (환불 추가) |
| **요구사항명** | 주문 상태 확장 (환불 상태) |
| **상태** | 유효 [추정] |
| **구분 / 우선순위** | 데이터(DAR) / 상(必) |
| **정의** | 주문 상태 체계에 환불 관련 상태값 3종(REFUND_REQUESTED 환불 요청, REFUND_PROCESSING 처리 중, REFUNDED 완료)을 추가하고 순차 자동 전이를 정의한다. |
| **상세 기능** | ① ORDERSTATUS 상태값 3종 추가 — REFUND_REQUESTED(환불 요청), REFUND_PROCESSING(처리 중), REFUNDED(완료)<br>② 순차 자동 전이 — REFUND_REQUESTED → REFUND_PROCESSING → REFUNDED 순서로만 자동 전이한다(Q-6 답변 '전자')<br>③ 기존 주문 데이터 마이그레이션 없음 — 신규 상태값은 신규 환불 건에만 적용한다 |
| **입력 데이터** | 환불 이벤트(요청 접수 / 처리 시작 / 처리 완료), 주문번호(orderId) |
| **출력 데이터** | ORDERSTATUS 상태값 갱신(REFUND_REQUESTED / REFUND_PROCESSING / REFUNDED) |
| **처리 흐름** | 환불 요청 접수 시 REFUND_REQUESTED 기록 → 자동 처리 시작 시 REFUND_PROCESSING 갱신 → 처리 완료 시 REFUNDED 갱신 |
| **선행 요구사항** | SFR-030 (주문 환불 기능) |
| **예외 / 제약** | · 현행 상태값 코드 그룹/enum 이 미정의(policy-domain-order.md §3)이므로 신규 값과 기존 저장값의 충돌 여부를 상세 설계에서 확인한다<br>· 역방향 전이(REFUNDED → REFUND_PROCESSING 등)는 허용하지 않는다<br>· 기존 주문 데이터 마이그레이션은 범위 외 — 기존 상태값 데이터를 변경하지 않는다 |
| **인수 기준(완료조건)** | · 환불 상태값 3종의 저장·조회 성공률 100%<br>· 정의된 순서(REFUND_REQUESTED → REFUND_PROCESSING → REFUNDED) 외 전이 발생 0건<br>· 기존 주문의 상태 데이터 변경 0건 |
| **검증 방법** | 시험 (TC-DAR-040-01~03 [추정] — 상태값 저장·조회 / 전이 순서 / 기존 데이터 불변) |
| **출처 / 추적** | 선행 요구사항 SFR-030 파생 + [확인필요] 답변 Q-2('알아서 해줘' — 설계 위임)·Q-6('전자' — 3단계 전이 확정) / 정책: policy-domain-order.md §3(상태값 코드 그룹/enum 미정의) / 데이터: ORDERSTATUS 테이블(db-schema.json 실재 대조 통과) / TC-DAR-040-01~03 [추정] |
