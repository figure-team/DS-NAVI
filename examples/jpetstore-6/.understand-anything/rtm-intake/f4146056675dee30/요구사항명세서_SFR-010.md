## 2. 요구사항 상세 명세

| 항목 | 내용 |
|---|---|
| **요구사항ID** | SFR-010 |
| **소속 요청ID** | REQ-001 |
| **요구사항명** | 네이버 소셜 로그인 |
| **상태** | 유효 |
| **구분 / 우선순위** | 기능 / 상(必) |
| **정의** | 기존 아이디/비밀번호 로그인과 병행하여, 로그인 폼 내부에서 네이버 계정으로 인증·로그인할 수 있게 하는 기능(사용자 결정: 로그인폼 내부 배치 — Q-2). |
| **상세 기능** | ① 로그인 폼(Account.action?signon) 내부에 네이버 로그인 버튼을 노출한다.<br>② 버튼 클릭 시 네이버 OAuth2 인가요청(SIR-010)으로 진입하고, 콜백에서 인가코드를 수신한다.<br>③ SIR-010 결과로 확보한 네이버 프로필(이메일)로 기존 ACCOUNT 를 조회해 일치 계정이 있으면 DAR-010 연계 테이블에 매핑을 저장하고 자동 연동한다.<br>④ 인증 성공 시 기존 로그인 성공 처리(AccountActionBean.java:169 계정 세션 등록)와 동일한 방식으로 세션을 구성한다.<br>⑤ 인증 실패(거부·오류) 시 기존 로그인 실패 처리(AccountActionBean.java:164,167 오류 메시지 후 SIGNON 복귀)와 동일하게 처리한다. |
| **입력 데이터** | 사용자의 네이버 로그인 버튼 클릭 이벤트, 네이버 OAuth 콜백(인가코드), SIR-010 이 반환하는 프로필(이메일·닉네임) |
| **출력 데이터** | 로그인 성공 시 계정 세션(기존 signon 성공과 동일 구조), 실패 시 오류 메시지 + 로그인 화면(SIGNON) 재표시 |
| **처리 흐름** | 로그인 화면 표시(AccountActionBean.java:151) → 네이버 로그인 버튼 클릭 → SIR-010 인가요청/콜백 → 인가코드→액세스토큰 교환·프로필 조회(SIR-010) → 이메일로 ACCOUNT 조회 → 일치 시 DAR-010 연계 테이블 매핑 후 기존 세션 등록 로직 재사용(AccountActionBean.java:161,169) → 불일치 시 신규 계정 생성 절차는 별도 정의 필요([확인필요] 범위 밖) → 실패 시 오류 메시지 표시 후 SIGNON 복귀(AccountActionBean.java:164,167) |
| **선행 요구사항** | - |
| **예외 / 제약** | 네이버 인증 거부·오류 응답 / 네이버 프로필에 이메일이 없거나 비공개 동의된 경우 / 이메일이 기존 ACCOUNT 복수 건과 일치하는 경우(email 유니크 여부 미확인, [확인필요]) |
| **인수 기준(완료조건)** | ① 로그인 폼 내부에 네이버 로그인 버튼이 노출되고 클릭 시 네이버 인증 플로우로 진입한다(AC-1). ② 네이버 인증 성공 시 기존과 동일한 세션 구성으로 로그인 처리가 완료된다(AC-2). ③ 네이버 인증 실패 시 기존과 동일한 오류 메시지와 함께 로그인 화면으로 복귀한다(AC-3). ④ 이메일이 일치하는 기존 계정이 있으면 신규 계정 생성 없이 자동 연동된다(AC-4). |
| **검증 방법** | 시험(신규 테스트케이스 정의 필요, [추정]) — 버튼 노출/클릭, OAuth 콜백 성공/실패, 기존 계정 자동 연동 3개 시나리오 |
| **출처 / 추적** | 고객 요청 REQ-001("네이버 로그인 기능 추가") / AccountActionBean.java:149,151,161,163,164,167,169 · after-flow.json(domain:account) / 시험케이스 미정([추정]) |
