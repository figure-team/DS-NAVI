# 한국어 산출 문체 규약 (ko-prose)

LLM이 쓰는 **모든 한국어 산출 문구**(도메인/흐름/단계 요약, 업무 흐름도 라벨, 정책 진술,
화면 설명·범례, 요구사항 문서, 영향 분석 서술)에 적용한다. 근거·신뢰도 규약(인용 verbatim,
`[확정]`⇒근거)은 각 스킬의 계약이 우선이며, 이 문서는 **문장 표면(문체·용어 표기)** 만 다룬다.

> 로드 규약: 프로젝트 override `.understand-anything/templates/style/ko-prose.md` → 없으면
> 플러그인 동봉본(이 파일). 사람이 편집하면 재빌드 없이 다음 실행부터 반영된다.

## 1. 종결·문형

- **표 셀·라벨·개조식 항목 = 명사형 종결.** "재고 확인 후 차감" ○ / "재고를 확인하고 차감합니다" ✕
- **산문 문장 = "~한다/~된다" 평서문.** "~합니다/~입니다" 존댓말 금지(문서체 통일).
- **이중 피동 금지.** "처리되어진다" ✕ → "처리된다" ○
- 한 산출물 안에서 종결 형태를 섞지 않는다(같은 표의 셀은 전부 명사형이거나 전부 평서문).

## 2. 문장 구성

- **한 문장 = 한 주장.** 60자를 넘으면 나눈다. 사실 두 개를 "~하며, ~한다"로 잇지 않는다.
- 대상(주어)이 바뀌면 주어를 다시 쓴다 — 주어 생략 연쇄로 "누가"가 모호해지면 안 된다.
- 나열이 3개 이상이면 문장에 우겨넣지 말고 개조식으로 푼다.

## 3. 번역투 금지 (자주 나오는 패턴)

| 금지 표현 | 대체 |
|---|---|
| ~에 대하여 / ~에 대한 | ~을/를, ~의 |
| ~를 수행한다 / ~를 진행한다 | 구체 동사(검증한다·저장한다·전송한다·조회한다) |
| ~하는 것을 가능하게 한다 | ~할 수 있게 한다 |
| 성공적으로 ~한다 | (삭제 — 성공 여부는 사실만) |
| ~됨을 확인할 수 있다 | ~된다 |
| ~에 의해 ~된다 | ~이/가 ~한다 (능동) |
| ~적인 측면에서 / ~와 관련하여 | ~에서 / ~은(는) |
| 해당 ~ (지시 대상이 명확할 때) | 그 ~, 또는 명사 반복 |

## 4. 용어 표기

- **산문에 코드 심볼 금지.** "checkInventory()" ✕ → "재고 확인" ○. 심볼이 꼭 필요하면
  괄호 병기 — "장바구니(Cart)".
- **업계 표준 외래어는 허용**(API, DB, URL, 로그인, 세션). 표준 번역이 있는 일반 단어를
  음차·원어로 남기지 않는다 — "페치한다" ✕ → "가져온다" ○, "밸리데이션" ✕ → "검증" ○.
- **같은 대상 = 같은 표기.** 한 산출물에서 "계정/어카운트/Account"처럼 흔들지 않는다.
  처음 등장 시 한 번만 병기한다. **기준 표기의 우선순위**:
  1. `.understand-anything/templates/style/ko-terms.md` — 사용자 확정 용어(있으면 최우선).
  2. `.understand-anything/doc-output/policy-glossary.md` — 코드 유래 용어집(DB 주석·enum).
  3. 둘 다 없으면 DB 컬럼 주석·기존 산출물 용어를 따른다.
  용어집은 **표기 기준**일 뿐 근거(인용)가 아니다 — citations 로 쓰지 않는다.
- 숫자+단위는 붙여 쓴다(8자, 10만원, 30일). 조사는 앞 단어의 한글 표기 기준으로 고른다.

## 부록 — 사용자 확정 용어 파일(ko-terms.md) 작성법

프로젝트의 `.understand-anything/templates/style/ko-terms.md` 에 표 하나로 쓴다(**선택 사항** —
없어도 전 파이프라인이 동작하며, 만들면 모든 산출물의 표기가 이 파일을 따른다):

```markdown
| 기준 표기 | 금지/혼용 표기 | 비고 |
|---|---|---|
| 계정 | 어카운트, Account | 화면 라벨은 "계정" 고정 |
| 주문 | 오더, Order | |
```

## 5. 좋은/나쁜 예 (few-shot)

**도메인 요약**
- ✕ "이 도메인은 주문에 대한 처리를 수행하는 기능들을 포함하고 있습니다."
- ○ "상품 주문의 생성·조회·취소를 담당한다. 결제 도메인과 주문 확정 시점에 연동된다."

**업무 흐름도 라벨** (명사형, ~30자)
- ✕ "사용자가 로그인을 시도하게 됩니다"
- ○ "로그인 시도"

**정책 진술**
- ✕ "패스워드에 대해 최소 8자 이상이어야 함이 요구되어진다."
- ○ "비밀번호는 최소 8자다."

**화면 설명(범례)**
- ✕ "이 버튼은 submitOrder() 함수를 호출하여 주문 프로세스를 진행하는 역할을 수행한다."
- ○ "주문을 확정하고 주문 완료 화면으로 이동한다."

**요구사항 정의**
- ✕ "네이버 로그인 기능과 관련하여 사용자가 소셜 계정으로 인증하는 것을 가능하게 한다."
- ○ "네이버 계정으로 로그인할 수 있다. 기존 아이디/비밀번호 로그인과 병행한다."
