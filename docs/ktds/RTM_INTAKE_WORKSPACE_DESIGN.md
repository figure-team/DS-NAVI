# RTM 인테이크 워크스페이스 설계서 — 모달+드로어 → 본문 세션 원장

> "＋ 새 요청"의 **모달(단계선택) → 스테퍼 + 하단 드로어(52vh)** 구조를 **추적표 "요청 세션" 탭 +
> 좌 270px 세션 원장 + 우 단계 콘텐츠**로 재설계한다. 부수로 **orphan 세션 결함**을 해소한다.
>
> **선행 설계:** `RTM_STEP_FLOW_DESIGN.md`(6단계 게이트 — **본 문서가 §7 UI 를 대체**),
> `RTM_TAB_DESIGN.md`(추적표 탭), `RTM_IMPACT_GATE_DESIGN.md`(근거 6축 — ②의 영향분석 표시가 맞물림),
> `FRONT_REDESIGN_DESIGN.md`(IA·URL 관례)
>
> **상태:** 위치·레이아웃은 사용자 확정(2026-07-16). 구현 범위(§5)는 승인 전.

---

## 0. 배경 — 왜 지금이 어색한가

사용자 지적: *"새 요청 버튼 누르면 단계 고를 수 있고, 분석 진행 시 아래 영역이 올라오며 결과를
보여주는데 이것보단 그냥 본문으로 빼는 게 낫다."*

**코드가 그 직관을 뒷받침한다.** 드로어 3종은 전부 "표에서 고른 한 행의 상세"이고 본문 기준
`absolute` 이며 `selXxx` 선택에 종속된다:

| 컴포넌트 | 형태 | 묶인 대상 |
|---|---|---|
| `FunctionDrawer`(48vh) · `RequirementDrawer`(50vh) · `ScenarioDrawer`(44vh) | `absolute` 드로어 | **선택된 행** |
| **`IntakeStepPanel`(52vh)** | **`fixed`** (`IntakePanel.tsx:100`) | **세션 수명** |

`IntakeStepPanel` 만 `fixed` 이고 선택이 아니라 **진행형 세션**에 묶여 있다 — 형태만 드로어이고
성격이 다르다. **이것이 어색함의 구조적 원인이다.** 다단계 작업은 "고른 한 행의 상세"가 아니다.

## 1. 확인된 사실 (2026-07-16 실측)

### 1.1 ★ orphan 세션 — 재설계가 고치는 실제 결함

**모델은 이미 복수 세션을 지탱한다**: `<sid>/` 디렉터리 격리, `isValidSid` path traversal 방어
(`vite.config.ts:1363`), `latestRtmSession()` 이 `readdirSync` 로 **여러 디렉터리를 순회**
(`vite.config.ts:1402-1417`). **그런데 UI 계약만 단일 슬롯**이라 반환은 1개다.

결과: **미완 세션 A 가 있어도(실행 중만 아니면) "＋ 새 요청"으로 B 를 만들 수 있다.**
`handleRtmIntakePost` 의 신규 분기(`vite.config.ts:1561-1570`)는 기존 세션 존재를 **확인조차
하지 않는다**. 그 순간 `startIntake` 가 `setSid(B)` 로 덮어쓰고(`useIntake.ts:47`)
`latestRtmSession()` 도 B 를 반환하므로 — **A 는 디스크에 남지만 UI 에서 영영 접근 불가**해진다.

- **세션 목록 엔드포인트: 없음**(`sessions` 문자열 0건). impact 는 `/impact-history` + `ledger.json`
  원장이 있으나 rtm-intake 는 **원장 자체가 없다**.
- **정리·상한: 없음**. `rtm-intake` 에 `rmSync`/`unlinkSync`/개수 상한 0건(impact 는
  `IMPACT_HISTORY_MAX = 50`). 설계문서도 미결로 인정 — `RTM_STEP_FLOW_DESIGN.md:330`.

### 1.2 IA 관례 — 탭은 렌즈(명사), 메뉴는 산출물 1:1

**탭 = 같은 데이터의 전치(transpose)** — `RTM_TAB_DESIGN.md:108` 이 명문화. 11개 메뉴의 모든 탭 중
**작업(동사) 축으로 갈린 탭은 0건**(기능/요청/시나리오/현황, 테이블/ERD/CRUD, 지도/구조/기능 …).

**메뉴 = 별도 산출물 파일 1:1** — 11개 메뉴가 전부 대응(rtm.json / impact.json / db-schema.json /
program-inventory.json / policy-signals / screens.json …). **인테이크는 고유 산출물이 없다** —
`rtm-intake/<sid>/` 는 rtm.json 으로 흘러드는 중간물이고 종착지가 요청 기준 탭이다
(`RtmView.tsx:118`).

### 1.3 라우트 분리는 이미 두 번 반려됐다

- `FRONT_REDESIGN_DESIGN.md:57` 이 `/rtm/intake/:sid` 를 규정 → **`:290` 이 뒤집음**:
  *"패널 상태라 별도 라우트면 RtmView 가 리마운트되어 뷰 상태가 소실됨"* → 쿼리로 결정.
- 업무지도도 같은 이유로 라우트 → 탭 재통합(`WorkMapTabs.tsx:6-9`: *"같은 페이지라 탭 전환 시
  언마운트가 없다"*).

### 1.4 원하는 화면은 이미 검증돼 있다

`lg:grid-cols-[270px_minmax(0,1fr)]` gap 14 가 **4곳에 리터럴 동일**:
`ChangeImpactView.tsx:710,776` · `DocsView.tsx:453` · `ScreenSpecView.tsx:802` ·
`data-map/TablesTab.tsx:373`. **공통: 좌 = 동종 항목의 원장/목록, 우 = 선택된 1건의 전체.**

진행형 상태의 URL 선례 — `ChangeImpactView.tsx:369-370`: `?run=<jobId>` 로 과거 job 스냅샷 열람,
좌측이 원장, 없으면 최신 live. **추적표는 이미 `?sid=` 를 미러 중**(`RtmView.tsx:148`).

| 신규 메뉴(`/intake`) | 반려 — 메뉴=산출물 1:1 기준 미충족(§1.2), 라우트 분리 반려 이력(§1.3) |

**남은 마찰(인정)**: 기존 4탭은 rtm.json 렌즈인데 세션은 다른 데이터다. 다만 **"요청"이라는 같은
명사의 생애주기 두 단계**(진행 중 → 반영 완료)로 읽으면 이어지며, 탭 라벨을 작업("새 요청")이
아니라 **대상("요청 세션")**으로 두어 렌즈 관례를 지킨다. 헤더의 `＋ 새 요청` 버튼은 **액션
어포던스로 유지**한다.

  목표 단계 칩은 6개, 기본 ⑥.

### 2.3 영향분석은 어디에 — 한 번 돌리고 두 곳에서 본다

인테이크가 영향분석을 돌리면 "그걸 어디에 보여주나"가 생긴다. **거짓 딜레마다** — 같은 데이터가
두 목적을 섬긴다:

> **갱신(2026-07-16)**: 영향분석은 초판이 상정한 ①식별 **안**이 아니라 **독립 단계 ②**다
> (`RTM_IMPACT_GATE_DESIGN.md` §6.5 — 6단계 승격). 아래 "한 번 돌리고 두 곳에서 본다" 결론은
> 그대로다 — 오히려 ②라는 **전용 표면**이 생겨 인라인의 자리가 더 분명해졌다.

| 표면 | 목적 | 시점 |
|---|---|---|
| **워크스페이스 ② 본문** | *이 요청을 컨펌할지 판단하기 위한 근거* | ② 컨펌 직전 — 결정하는 자리 |
| **변경·영향 메뉴** | *지금까지 돌린 분석들의 열람* | 사후, 원장에서 |

→ **인테이크의 impact 실행을 impact 원장(`impact-history/`)에 기록**하고(query = 요청 원문),
워크스페이스는 ②의 본문으로 렌더한다. 이러면 `RTM_IMPACT_GATE_DESIGN.md §10-4`(원장 노출 여부)와
`§C3`(요청별 저장소)이 동시에 풀리고, `/change` 의 "기록 없는 분석" 고아 표시도 안 생긴다.

**워크스페이스가 통째로 변경·영향에 갈 수 없는 이유가 여기 있다** — 워크스페이스는 영향분석(②)과
문서(③④⑤)를 **둘 다** 담는 그릇이고, 변경·영향은 그중 하나를 보는 렌즈다.
