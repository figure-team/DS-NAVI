## 1. 한 줄로

> ### "레거시 프로젝트에 명령 한 번이면, **코드 근거가 붙은 운영 문서**가 자동으로 만들어집니다."

Understand-Anything(오픈소스) 기반 **Claude Code 플러그인**. 별도 프로그램 설치 없이, 쓰던 AI CLI 명령으로 바로 사용합니다.

> 📌 이번 **MVP는 "근거 기반 문서 자동화 + 검토/승인"** 을 증명하는 단계입니다. **민감정보 보호(보안 게이트)는 다음 단계(Phase 2)** 로, 실고객 적용 전에 추가됩니다.

---

## 2. 어떤 문제를 푸나요? (Before)

레거시 유지보수 현장의 반복되는 고통:

- 📄 **문서가 코드와 따로 논다** — 마지막 업데이트가 언제인지 모른다.
- 🧑‍💻 **담당자가 떠나면 깜깜이** — 시스템 구조가 사람 머릿속에만 있다.
- 🗄️ **PL/SQL·MyBatis는 미궁** — 화면-API-테이블 연결을 손으로 추적한다.
- 🆕 **온보딩에 수주~수개월** — 신규 투입 인력이 코드를 처음부터 읽는다.
- 🔐 **(다음 단계 과제)** 문서 작성 중 설정 파일의 비밀번호·주민번호가 그대로 노출되기도 한다.

## 3. 어떻게 바뀌나요? (After)

| 항목 | Before | After | 단계 |
| --- | --- | --- | :---: |
| 신규 투입 온보딩 | 수주~수개월 | **1~2일** | MVP |
| 문서 최신화 | 수동, 누락 잦음 | **실행할 때마다 자동 갱신** | MVP |
| PL/SQL 파악 | 담당자 의존 | LLM 요약 + `[추정]` 태그 | MVP |
| 문서 신뢰성 | 보증 없음 | 검토·승인·감사 내재화 | MVP |
| 변경 영향도 파악 | 수동 코드 추적 | `/understand-impact` 자동 분석 | MVP+ |
| 민감정보 유출 | 작성 중 실수 가능 | 보안 게이트 자동 차단 | Phase 2 |

## 4. 핵심 가치 — MVP 2가지 + 다음 단계 1가지

### ① 지어내지 않습니다 — 근거 기반 〔MVP〕
모든 문장에 `근거: 파일경로:라인` 이 붙습니다. 근거가 없으면 **저장 자체가 차단**됩니다.

> 예) `LoginController는 /login 요청을 처리한다. [확정(AI)] 근거: LoginController.java:42`

### ② 믿을 수 있습니다 — 검토·승인·감사 〔MVP〕
AI가 쓴 초안(DRAFT)을 담당자가 검토·승인(APPROVED)하고, 모든 과정이 감사 로그로 남습니다.

### ③ 새어나가지 않습니다 — 보안 게이트 〔Phase 2 / 다음 단계〕
LLM에 보내기 **전**에 비밀번호·주민번호·토큰을 분류·마스킹하고, 저장 **전**에 한 번 더 검사합니다.

> 실고객(특히 폐쇄망) 적용을 위한 **다음 단계 핵심 기능**. MVP에서는 비민감 샘플로 ①·②를 먼저 증명합니다.

## 5. 어떻게 동작하나요? (한 장 요약)

```
[레거시 코드]
   │  /understand-docs
   ▼
┌─────────────────────────────────────────────┐
│ ① AI 분석   지식 그래프 → 근거 매핑 → 문서화  │   ◀ MVP
│    코드 근거가 붙은 문장만 통과               │
└───────────────┬─────────────────────────────┘
                ▼
[근거 붙은 문서 5종]  →  담당자 검토·승인  →  APPROVED   ◀ MVP


   ┌─ Phase 2에서 앞뒤로 추가되는 보안 게이트 ──────────┐
   │ (앞) 입력 검사  파일 4등급 분류·마스킹, SECRET 제외 │
   │ (뒤) 출력 재검증  민감정보 재스캔 → 통과해야 저장   │
   └────────────────────────────────────────────────┘
```

> MVP 흐름은 위쪽(분석→문서→승인). 보안 게이트는 Phase 2에서 이 흐름의 앞(입력)과 뒤(출력)에 끼워집니다.

## 6. 유스케이스 ① — 신규 투입 인력 온보딩 〔MVP〕

**상황**: 처음 보는 Spring/MyBatis 샘플 프로젝트에 막 투입됐다.

```
$ /understand-init          # 프로젝트 초기화 (VCS 자동 감지)
$ /understand-docs          # 5종 문서 자동 생성
```

**결과 — `docs/` 폴더에 즉시 생성**

| 문서 | 보는 것 |
| --- | --- |
| `01_tech-stack.md` | 어떤 언어·프레임워크·라이브러리를 쓰나 |
| `02_architecture.md` | 레이어 구조와 모듈 의존 관계 |
| `03_feature-spec.md` | 어떤 업무 기능이 있고 어디서 시작되나 |
| `04_api-spec.md` | 어떤 API가 있고 어느 Controller가 처리하나 |
| `05_db-spec.md` | 어떤 테이블·컬럼이 있고 어느 Mapper가 쓰나 |

> 코드를 처음부터 읽는 대신, **근거 링크를 따라가며** 구조를 파악합니다.

## 7. 유스케이스 ② — 문서 신뢰성 (검토·승인) 〔MVP〕

**상황**: AI가 쓴 문서를 공식 운영 문서로 채택하기 전 검증이 필요하다.

```
$ /understand-docs review --list                      # 검토 대기 + [추정] 항목 수
$ /understand-docs review --doc 04_api-spec.md         # [추정] → [확정(홍길동)] 확정
$ /understand-docs approve --doc 04_api-spec.md --by "홍길동"   # 공식 승인
```

- `[확정(AI)]` / `[확정(담당자)]` / `[추정]` / `[확인 필요]` 4단계 신뢰도 태그
- 동적 SQL·리플렉션 등 자동 판단이 어려운 항목은 `[확인 필요]`로 남겨 **담당자에게 위임**
- 생성·검토·승인 전 과정이 감사 로그에 기록

---

## 8. 유스케이스 ③ — 변경 영향도 분석 〔MVP+〕

**상황**: "로그인에 간편 로그인을 추가하면 어디까지 영향이 갈까?"

```
$ /understand-impact 로그인 기능에 간편 로그인 추가        # 자연어
$ /understand-impact --path src/auth/LoginService.java   # 파일 지정
```

**분석 범위**: API 영향 / DB 영향 / 업무 흐름 영향 / 연관 모듈
**출력**: `docs/09_release/change-impact-analysis.md`

> 수정 **전·후** 모두 영향 범위를 미리 보여줍니다.

---

## 9. 유스케이스 ④ — 민감정보 보호 (보안 게이트) 〔Phase 2〕

**상황**: `application-prod.yml` 에 DB 비밀번호가, 테스트 데이터에 주민번호가 들어있다.

```
$ /understand-docs
  → application-prod.yml ........ [SECRET]    LLM 전송 제외
  → CustomerService.java ........ [INTERNAL]  구조만 전송, 값은 <REDACTED>
  → UserFixture.sql ............. [SENSITIVE] 주민번호 패턴 감지 → 로컬만
```

- 한국 특화 PII(주민·카드·계좌·전화·사업자번호)와 토큰·키를 **맥락+구조+무작위도** 조합으로 탐지
- 보안 심의용 **Transfer Manifest**(어떤 파일이 어떻게 처리됐는지 요약) 자동 생성

> **실고객·폐쇄망 적용을 여는 다음 단계 기능.** "외부로 무엇이 나갔는가"를 문서로 증명합니다.

## 10. 데모 시나리오 (MVP 전체 흐름)

```
1. /understand-init        → 프로젝트 인식, 설정 생성
2. /understand             → 지식 그래프 생성 (U-A)
3. /understand-docs        → 근거 붙은 5종 문서 (DRAFT)
4. /understand-docs review → [추정] 항목 검토·확정
5. /understand-docs approve → 공식 승인 (APPROVED)
6. /understand-export      → HTML 산출물
```

각 문서 문장마다 **근거**가, 각 단계마다 **감사 로그**가 자동으로 따라붙습니다. (보안 게이트는 Phase 2)

---

## 11. 단계별 구현 범위

**✅ 이번 MVP (유형3 개방형 — Claude Code 기준, 비민감 샘플)**

- `/understand-init`, `/understand-docs`, `/understand-export`
- 근거 붙은 5종 문서 생성
- DRAFT → 검토 → 승인 → 감사 로그

**🔜 MVP+ (안정화 후)**

- `/understand-impact` 변경 영향도, `/understand-history` 이력, `/understand-custom-dashboard`, `/understand-export` PPT/Word 추가 포맷

**⏭️ Phase 2 (실고객 적용 전 필수)**

- **보안 게이트 전체**(입력 검사·출력 재검증·중계 검사), PII/secret 탐지, prompt injection 방어, Transfer Manifest
- 유형1·2(폐쇄망 로컬 LLM / 자사 중계)

**⏭️ 후속 트랙 (별도 계약/프로젝트)**

- 자사 중계 게이트웨이 실제 운영, 운영 DB 메타데이터, 커스텀 대시보드 고도화

## 14. 정리

- 레거시에 **근거 기반 문서 + 검토/승인**을 입히는 자동화 도구입니다.
- 별도 CLI 없이 쓰던 AI CLI 명령으로, 오픈소스를 최대한 보존하며 확장합니다.
- **MVP**: 비민감 샘플에서 **①근거 + ②신뢰성** 두 축을 증명. **Phase 2**: 보안 게이트로 실고객·폐쇄망까지 확장.

> 다음 단계: 비민감 샘플 2종 선정 → 필수 MVP 구현 → 릴리즈 게이트 통과 → **Phase 2 보안 게이트 → 고객사 PoC.**
