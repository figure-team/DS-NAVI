## 2. 요구사항 상세 명세

| 항목 | 내용 |
|---|---|
| **요구사항ID** | SER-010 |
| **소속 요청ID** | REQ-001 |
| **요구사항명** | 소셜 로그인 세션 보안 |
| **상태** | 유효 |
| **구분 / 우선순위** | 보안 / 중 |
| **정의** | 네이버 로그인 도입에 따라 기존 인증 상태 판정·세션 관리 원칙을 동일하게 적용하고, 네이버 액세스 토큰을 세션에 원문 노출하지 않도록 보호하는 보안 요구사항. |
| **상세 기능** | ① 네이버 로그인 성공 후에도 기존 인증 상태 판정 규칙(인증 표시·계정 정보·아이디 보유)을 동일하게 적용한다.<br>② 네이버 액세스 토큰은 프로필 조회 후 세션에 원문으로 저장하지 않는다. |
| **입력 데이터** | SIR-010 이 발급받은 네이버 액세스 토큰(임시), 로그인 성공 후 세션 객체 |
| **출력 데이터** | 기존과 동일한 보안 원칙이 적용된 세션 상태(토큰 미포함) |
| **처리 흐름** | SIR-010 에서 액세스 토큰으로 프로필 조회 → 조회 완료 즉시 토큰 폐기(세션 미저장) → SFR-010 의 기존 세션 등록 로직 재사용(AccountActionBean.java:169, 비밀번호 미저장과 동일 원칙) |
| **선행 요구사항** | SFR-010 |
| **예외 / 제약** | 토큰이 로깅·예외 스택 등 부수 경로로 노출될 위험 / 세션 고정(session fixation) 공격 가능성 |
| **인수 기준(완료조건)** | ① 로그인 성공 후 세션 덤프에서 네이버 액세스 토큰 평문이 발견되는 건수 0건. ② 네이버 로그인 사용자의 인증 상태 판정이 기존 로그인 사용자와 동일하게 동작. |
| **검증 방법** | 검토(코드 리뷰) — 세션 저장 객체 필드에 토큰 미포함 확인 |
| **출처 / 추적** | SFR-010(네이버 소셜 로그인)에서 파생 / AccountActionBean.java:169 / 시험케이스 미정([추정]) |
