# 구현 기능 상세 (Features)

> **제품:** Understand Anything (code-atlas) — 코드베이스를 인터랙티브 지식 그래프로 변환하는 분석·시각화 플랫폼
> **기준 버전:** U-A `2.7.7-ktds.9` · ktds-legacy `0.14.5`
> **검증 대상:** JPetStore-6 (Spring/MyBatis 레거시 자바 쇼핑몰)

이 문서는 **현재까지 실제로 구현·동작하며, 실데이터로 산출물 생성이 검증된** 기능만을 다룬다. (★ = KTDS가 설계·구축한 영역)

---

## 1. 동작 원리 — 4단 파이프라인

코드베이스는 다음 4단계를 거쳐 지식 그래프(JSON)로 변환되고, 대시보드에서 시각화된다.

```
코드베이스
  │
  ① 정적분석 (tree-sitter)   ─ 구조 100% 결정적 확정 (파일·클래스·함수·호출 관계)
  │
  ② LLM 의미 보강            ─ "무엇을 하는지" 요약·도메인·이름 채움 (파일:라인 인용 강제)
  │
  ③ 기계 검증               ─ 인용을 실제 소스와 대조, 불일치는 [확인 필요]로 강등 (환각 차단)
  │
  ④ 지식 그래프 → 대시보드    ─ JSON 산출물을 커밋해 팀이 공유, 4대 기능으로 시각화
```

**핵심 설계 원칙**

- **결정적 우선(Deterministic-first).** 노드와 호출 관계 같은 *구조*는 정적분석이 확정한다. LLM은 *의미*(이름·요약·도메인 경계)만 채운다 → 같은 커밋이면 재현 가능.
- **인용 강제 + 기계 검증.** 모든 LLM 설명에는 `{filePath, line, snippet}` 인용이 1개 이상 필수이며, emit 단계에서 실제 소스 파일과 대조한다. 경로·라인·텍스트가 맞지 않으면 삭제하지 않고 `[확인 필요]`로 강등 → 환각이 결과에 침투하지 못한다.
- **산출물 = 커밋 가능한 자산.** 분석 결과는 `.understand-anything/` 아래 JSON으로 저장된다. 한 번 분석해 커밋하면 팀 전원이 재사용하고, 변경분만 증분 갱신한다.

---

## 2. 산출물 한눈에

| 기능 | 산출물 파일 | 생성 스킬 |
|---|---|---|
| 구조 그래프 | `knowledge-graph.json` | `/understand` |
| 도메인 지도 ★ | `domain-graph.json` | `/understand-map` |
| 변경 영향도 ★ | `impact-overlay.json` | `/understand-impact` |
| 문서/위키 ★ | `wiki-graph.json` (+ `wiki-meta.json`) | `/understand-docs` |
| 증분 갱신 | `fingerprints.json` | (공통) |

대시보드는 디렉터리에 존재하는 산출물을 감지해 해당 뷰 모드를 활성화한다.
뷰 모드: `structural`(구조) · `domain`(도메인) · `wiki`(문서) + `영향도` 오버레이.

## 3. 기능별 상세

### ① 구조 그래프 (Structural)

**무엇을 하나.** 전체 코드베이스를 스캔해 파일·클래스·함수를 노드로, import·호출·의존을 엣지로 갖는 아키텍처 그래프를 만든다.

**구현 핵심**
- **tree-sitter(WASM) 정적분석**으로 함수/클래스 정의, import/export, 상속, 호출 지점을 결정적으로 추출.
- **노드 13종 × 엣지 35종** 스키마(Zod 검증, LLM 별칭 자동 정규화).
- **아키텍처 레이어 자동 분류** — 파일 경로 + import 그래프 + 노드 요약으로 3~10개 레이어를 식별, 색상으로 구분.
- **증분 갱신** — 콘텐츠 해시 기반 변경 감지(`fingerprints.json`)로 바뀐 파일만 재분석.

**대시보드(structural 뷰)**
- 75% 그래프 + 우측 사이드바(Info/Files 탭) 레이아웃.
- 노드 클릭 → 하단에서 슬라이드업되는 **소스 코드 뷰어**(접근 토큰 + 경로 화이트리스트로 게이팅).
- 이름/시맨틱 **검색**, 레이어 범례.

---

### ② 도메인 지도 (Domain) ★KTDS 핵심

**무엇을 하나.** 코드를 **업무 도메인 단위**로 재구성하고, 각 도메인의 **업무 흐름**과 그 흐름이 실제로 거치는 **실행 경로**(API → Service → DAO → DB)를 시각화한다.

#### 2-1. 엔진 — 결정적 도메인 추출 (`/understand-map`)

`스캔 → 휴먼 게이트 → LLM 보강 → 기계 검증 → emit` 파이프라인.

- **결정적 스켈레톤** — 라우트/API 진입점, 호출 체인, 도달성을 정적으로 계산하고 도메인 경계 후보를 도출(`.spec/map/` 하위 census/routes/edges/slices/candidates).
- **휴먼 게이트(S7)** — `confirm` 세션에서 사람이 도메인 경계를 **승인/이름변경/병합/제외**. (LLM·사람 도메인 경계 합의 상한 ~56% 문제를 사람이 끊어줌)
- **LLM 보강(S8)** — 도메인/흐름/스텝의 이름·요약·엔터티·업무규칙을 한국어로 채움. 구조는 read-only, **인용 필수**.
- **기계 검증(S9) → emit** — 인용을 소스와 대조해 `domain-graph.json` 생성. 멱등(실패분만 재실행 가능).

#### 2-2. 메서드 단위 호출 그래프 ★ (정확도의 핵심)

기존 도메인 흐름은 **파일↔파일 의존 투영**이었다 → 같은 협력 클래스를 2번 호출해도 1스텝으로 뭉치고 **실행 순서가 소실**됐다.
이를 **실제 메서드 호출 트레이스**로 교체했다.

- tree-sitter로 메서드 바디의 `method_invocation`을 전수 수집하고, **수신자 타입을 해소**한다: `this/super/field/param/local/static/chain(리턴타입 추론)`.
- **오버로드 해소** — 메서드를 이름만이 아니라 `(이름, 인자수)`로 키잉해 동명 오버로드 혼동을 제거.
- **호출그래프 기반 흐름 체인** — 핸들러부터 *실제 호출을 따라가* 도달한 파일만 흐름 스텝으로 구성(구조 인접 BFS 대체).
- 각 호출 엣지에 **호출 메서드를 순서대로** 라벨링(`description`) → 대시보드에서 노드별 사용 메서드로 표시.

> **효과:** JPetStore 기준 미해소 호출률 **36% → 6%**. 예) `editAccount` 흐름이 실소스 순서(`updateAccount → getAccount → getProductListByCategory`)대로 정확히 추출됨.

#### 2-3. 대시보드 — 3단 드릴다운

도메인 모드는 U-A 일반 그래프를 대체하는 **독립 전체 화면 경험**이다(ADR-006).

1. **도메인 지도(`DomainMapView`)** — 도메인 카드 랜딩 + 통계(도메인 수·흐름 수·스텝 수·언어/프레임워크).
2. **흐름 목록(`FlowListView`)** — **좌측 사이드바**에 흐름 목록(엔드포인트 전체 + 기능 라벨), 클릭 시 **중앙에 코드 그래프**, **우측 사이드바**는 노드 클릭 시에만 상세를 표시(닫기 버튼 제공).
3. **흐름 스파인(`FlowSpineView`)** — 한 흐름의 백엔드 호출 체인을 `API→Service→DAO→DB(+Other)` 레인에 **가로로** 배치, 실행 순서대로 교차계층 연속 엣지. (순수 JS 좌표 계산, ELK 미사용)

**스파인 표현 기법**
- **곁가지 접기(Branch folding)** — 핵심 백본(api/service/dao/db)은 항상 표시, 부가 데이터 엔터티("Other" 레인)는 접어두고 `＋N 데이터 엔터티` 배지로 펼침 → "큰 그림 먼저, 디테일은 나중에".
- **사용 메서드 펼침** — 노드 아래 `ƒ N methods` 칩, 펼치면 호출 메서드를 순서대로 나열하고 **아래 노드들을 밀어내** 겹침 없이 표시.

---

### ③ 변경 영향도 (Impact) ★KTDS

**무엇을 하나.** 변경한 코드가 **무엇에 영향을 주는지**를 커밋 전에 예측한다.

**구현 핵심**
- 변경 파일을 시드로 지식 그래프 위에서 **정방향/역방향 도달성**을 계산 → 영향받는 API·DB·흐름을 산출.
- 결과는 `impact-overlay.json`으로 저장되어 대시보드에서 **영향 노드 하이라이트** 오버레이로 표시.
- 결정적 분석(같은 커밋·diff면 동일 결과).

> **검증:** JPetStore에서 주문(Order) 관련 **5개 파일 변경 → 영향 11개 노드** 식별(상류/하류 파일 수, 미해소 항목 포함 메타 제공).

---

### ④ 문서/위키 (Docs) ★KTDS

**무엇을 하나.** 지식 그래프 + 도메인 그래프로부터 **사람이 읽는 문서**(README·기능명세·온보딩·엔터티 카탈로그 등)를 자동 생성한다.

**구현 핵심**
- 도메인/흐름/업무규칙/엔터티를 묶어 **기능명세**로 합성하며, **모든 문장에 `파일:라인` 출처**를 부착.
- 세그먼트형 위키(`wiki-graph.json` + `wiki-meta.json`)로 산출 → 대시보드 **문서 리더(wiki 뷰)**에서 메타데이터 + 본문 열람(ADR-004).

> **효과:** 문서화 비용 절감 + 모든 서술이 출처로 추적 가능 → 신뢰할 수 있는 온보딩 자료.

## 5. 기술 스택

- **분석 엔진(core):** TypeScript(ESM, strict), `web-tree-sitter`(WASM), Zod 스키마, Vitest.
- **도메인 엔진(ktds-legacy-core):** TypeScript, 결정적 스켈레톤 + 인용 검증, Vitest.
- **대시보드:** React + TypeScript, React Flow, Zustand, TailwindCSS v4, prism-react-renderer(소스 뷰어).
- **모노레포:** pnpm workspaces. 코어는 subpath export(`./search`·`./types`·`./schema`)로 브라우저에서 Node 모듈 미유입.
- **언어 지원:** 10여 개 프로그래밍 언어 + 설정/인프라 포맷(YAML·SQL·Dockerfile·Terraform 등).
