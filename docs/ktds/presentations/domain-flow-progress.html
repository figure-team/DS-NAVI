문제

레거시는 "한 기능이 어디까지 흐르는지"가 안 보인다

  • 코드는 API · Service · DAO · DB로 계층마다 흩어져 있는데, 한 요청이 그 계층들을 어떻게 관통하는지는 어디에도 그려져 있지 않다.
  • 기존 구조 그래프는 파일/클래스 단위 의존성만 보여줘 "기능 단위 흐름"과는 거리가 멀다.
  • Spring · Stripes 등 프레임워크마다 진입점 규칙이 달라 흐름의 시작점을 일관되게 잡기 어렵다.
  • 비개발자 / 신규 인력이 도메인 관점으로 시스템을 이해할 진입로가 없다.
접근 · ADR-006

#1 · 엔진 Layer Emit

step 노드가 직접 layer를 안다

예전엔 대시보드가 파일명 휴리스틱으로 레인을 추측 → jpetstore 패키지명 org.mybatis의 "mybatis" 토큰 때문에 전건 DB 오분류. 이제 엔진이 route/edge 근거로 api·service·dao layer를 직접 부착.

A · 이벤트별 흐름 분리

ActionBean 1개 → 핸들러별 흐름

Spring은 이미 메서드마다 흐름인데 Stripes만 클래스당 1개로 뭉쳐 있던 비일관 해소. 이벤트 핸들러마다 라우트 emit → Account 1 → 7 흐름.

B · 실제 calls 엣지

가짜 합성 순서가 아닌 진짜 호출

step→step 엣지를 실제 호출 토폴로지로 emit + 핸들러 bodyText에서 주입 필드를 추출해 핸들러별로 다른 step 체인 구성.

C · 곁가지 접기 + 파이프라인 안정화

노이즈 제거 · stale-dist 회귀 픽스

추상 베이스 클래스를 step에서 제외(루트 보존). ensure-built가 구버전 dist를 조용히 실행하던 회귀를 버전 스탬프 검사로 차단.
